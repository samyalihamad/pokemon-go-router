# PokéStop Route Optimizer — Plan & Design

A personal project that doubles as a **DS/Algo practice sandbox**: the infra + map UI
are built; **you implement the routing algorithms in Python** and watch them draw on a
real map of downtown Palo Alto (on your laptop *or* your iPhone).

This is graph + geometry + routing — the same shape as Waymo's world.

---

## The problem

Given ~72 PokéStops in downtown Palo Alto, find the best walking route. "Best" evolves:

- **Objective A — Optimal tour (TSP):** visit *every* stop in minimum total distance. Order matters.
- **Objective B — Orienteering:** visit *as many stops as possible* within a walking budget.
- **Objective C — Max farming rate:** maximize **spins per hour** on a repeating loop, respecting
  the **5-minute cooldown** (a stop re-arms 5 min after you spin it).

## The model

- **Nodes** = PokéStops.
- **Edges** = walkable connections (sidewalks — you can't cut through buildings).
- **Edge weight** = walking distance/time. (v1 uses straight-line `haversine`; the advanced
  stub `street_distance` upgrades this to true sidewalk-graph distance via Dijkstra.)

## Key results we derived (the "why" behind the stubs)

1. **Distance between two stops = shortest path** → **BFS** (unit edges) or **Dijkstra** (weighted).
   This is a *subroutine*, not the whole problem. Precompute an all-pairs matrix once.
2. **Objective A is the Traveling Salesman Problem (TSP)** — **NP-hard**. Brute force is `O(n!)`
   (80 stops ⇒ more orderings than atoms in the universe). Exact is only feasible for small `n`
   (Held-Karp DP, `O(2^n·n^2)`, ~≤15). For large `n` you use **heuristics/approximations**.
3. **Greedy nearest-neighbor is a fast heuristic but NOT optimal** — provable with a 4-stop
   counterexample where greedy strands a stop and backtracks. (Contrast: Dijkstra's greedy *is*
   optimal, because shortest paths have optimal substructure; TSP tours do not.)
4. **The cooldown inverts the intuition.** Spin rate `= k / max(T, C)`:
   - loop time `T < C` → you idle waiting for re-arm (cooldown-limited).
   - `T ≥ C` → walk-limited; stretching the loop just lowers the rate.
   - **Sweet spot: the densest cycle whose walk-time ≈ the cooldown.** So a *shorter* loop isn't
     always better — Objective C is an **orienteering-on-a-cycle** problem.
5. **Clustering tames the explosion.** Partition stops into clusters, solve small TSPs *inside*
   each (n! is fine when n≈10), route between cluster super-nodes. **Cluster-first-route-second**
   (Karp's partitioning). It trades the optimality guarantee for tractability — the classic
   engineering response to NP-hardness.

## Architecture

- **Frontend:** static HTML + [Leaflet](https://leafletjs.com) map (OpenStreetMap tiles). Responsive;
  installable to the iPhone home screen (PWA `manifest.json`).
- **Algorithms:** **Python**, run *in the browser* via **[Pyodide](https://pyodide.org)** (WASM).
  numpy is preloaded. No backend — "deploy" = `git push` (GitHub Pages) or just refresh locally.
- **Data:** `data/pokestops.json`, generated by `scripts/generate_pokestops.py` (synthetic PA grid).

```
index.html            UI shell (+ Pyodide, Leaflet, PWA)
algorithms.py   ★     YOUR practice file — infra done, 6 stubs to implement
js/app.js             boots Pyodide, loads algorithms.py, wires the buttons
js/map.js             Leaflet rendering (stops, routes, clusters)
data/pokestops.json   the stops
scripts/generate_pokestops.py
```

## Your practice roadmap (implement in this order)

Each function lives in `algorithms.py`. Return a list of stop indices (or `None` = unsolved).

| # | Function | Concept | Difficulty |
|---|----------|---------|------------|
| 1 | `nearest_neighbor_route` | Greedy heuristic | ⭐ warm-up |
| 2 | `optimal_tour` | Exact TSP (brute force / Held-Karp) — guard large n! | ⭐⭐⭐ |
| 3 | `max_stops_within_budget` | Orienteering (prize-collecting) | ⭐⭐⭐ |
| 4 | `best_cooldown_loop` | Max spins/hr cycle (uses the rate formula) | ⭐⭐⭐⭐ |
| 5 | `cluster_stops` | k-means / grid clustering (numpy available) | ⭐⭐ |
| ★ | `street_distance` | Sidewalk graph + Dijkstra → all-pairs matrix | ⭐⭐⭐ |

**Suggested path:** 1 (baseline) → 5 (clusters) → 2 (TSP inside a cluster) → 3 → 4 (the real game) → ★.

## How to run

**Locally** (edit-refresh loop):
```bash
python3 -m http.server 8000
# open http://localhost:8000
```
**On your iPhone:** open the GitHub Pages URL (see README), then Share → *Add to Home Screen*.
Edit `algorithms.py` → `git push` → refresh on your phone.
